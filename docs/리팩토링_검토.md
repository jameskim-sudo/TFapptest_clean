# 코드 리팩토링 검토 결과

## 2. App.tsx — 화면 전환 핸들러

### 현상
- **handleTabChange**와 **handleReportTabChange**가 로직이 완전히 동일함 (tab → screen 매핑만 함).
- `setCurrentScreen("...")`만 하는 핸들러가 15개 이상 있음.

### 제안
- 탭 전환용 핸들러를 **하나로 통합**하고, `handleReportTabChange`는 제거 후 `handleTabChange`만 사용.
- 선택 사항: `navigate(screen: Screen)` 하나로 두고, 단순 이동은 `navigate("home")` 형태로 호출하도록 정리.

### 효과
- App.tsx 라인 수 감소, 새 화면 추가 시 수정 지점 최소화.

---

## 3. StatusBar 중복

### 현상
- **app/components** 내 **13개 파일**에서 각각 `function StatusBar()`를 정의함.  
  (HomeScreen, RecordScreen, ReportListScreen, MyScreen, AnimatedOnboardingScreen, OnboardingScreen, RunningCoachingScreen, PilatesCoachingScreen, LoginScreen, LoadingScreen, GolfCoachingScreen, GoalSettingScreen, DashboardScreen)
- 시계·배터리·신호 아이콘 등 **구조는 동일**하고, SVG import 경로만 파일마다 다름 (Figma export 등으로 인한 차이).

### 제안
- **공통 컴포넌트** `StatusBar.tsx` 한 곳에 두고, 필요한 SVG는 한 세트만 사용하거나 공통 asset으로 두기.
- 각 화면에서는 `import StatusBar from "@/app/components/StatusBar"` 후 기존 로컬 `StatusBar` 정의 제거.

### 효과
- 상단 바 수정 시 한 곳만 변경하면 됨. 스타일/접근성 통일 용이.

### 주의
- Figma export SVG 경로가 파일마다 다르므로, 공통화 시 **한 가지 디자인의 SVG만** 쓰거나, StatusBar가 props로 SVG 경로를 받도록 설계해야 함.

## 4. 그 외

- **NavigationBar**: 이미 단일 컴포넌트로 분리되어 있음.
- **imageUrls.ts**: 설정이 한 곳에 모여 있어 적절함.
- **CoachToneContext**: 역할이 명확하고 단순함.
- **각 Screen 컴포넌트**: 화면별로 책임이 나뉘어 있어, 당장의 대규모 리팩터링은 필수는 아님.

## 5. 권장 적용 순서

1. **App.tsx**: `handleReportTabChange` 제거 후 `handleTabChange`만 사용하도록 통합. (변경 범위 작고 즉시 적용 가능)
2. **StatusBar**: 공통 `StatusBar.tsx` 추가 후, 사용처를 하나씩 교체. (변경 범위 큼, 단계별 진행 권장)

원하면 1번부터 구체적인 패치(코드 수정안)까지 이어서 정리할 수 있음.
