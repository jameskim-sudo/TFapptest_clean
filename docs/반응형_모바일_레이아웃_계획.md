# 휴대폰 반응형 레이아웃 개선 계획

## 1. 현재 현상 정리

- **증상**: 기기마다 본문·하단 네비게이션바가 **보이는 기기**와 **안 보이는 기기**가 있음.
- **목표**: 어떤 디바이스에서든 **전체 내용(본문 + 하단 네비)** 이 항상 보이도록 반응형 처리.

---

## 2. 원인 분석

### 2.1 뷰포트·높이

| 항목 | 현재 | 문제 |
|------|------|------|
| 앱 컨테이너 | `w-[393px] h-[852px]` + `maxHeight: 100dvh` | 높이가 `min(852px, 100dvh)`로 고정에 가깝게 동작. 짧은 기기에서 852px 기준 레이아웃이면 하단이 잘릴 수 있음. |
| 외부 래퍼 | `h-screen` (100vh) | 모바일에서 주소창 등으로 100vh와 실제 보이는 영역이 달라질 수 있음. |
| `index.html` | `viewport`만 설정 | `viewport-fit=cover` 없으면 노치/홈 인디케이터 영역이 뷰포트에 반영되지 않을 수 있음. |

→ **원인**: 고정 높이(852px) 의존 + 뷰포트 높이 단위 불일치 + safe area 미반영으로, 기기별로 “보이는 영역”과 레이아웃이 어긋날 수 있음.

### 2.2 레이아웃 구조

| 항목 | 현재 | 문제 |
|------|------|------|
| 본문 | `absolute top-[96px]` 등으로 배치 | 하단 여백이 없어, 짧은 화면에서 네비 영역과 겹치거나 잘림. |
| 네비게이션바 | `position: fixed; bottom: 0` | 뷰포트 기준 고정이라, 컨테이너가 짧을 때 컨테이너 밖으로 나가 보이지 않을 수 있음(기기/브라우저마다 다름). |
| 컨테이너 | `overflow: hidden` | 넘치는 내용(본문+네비)이 잘림. |

→ **원인**: “본문 영역”과 “네비 영역”이 하나의 **유연한 컬럼 레이아웃**으로 정해져 있지 않아, 기기 높이에 따라 하단이 잘리거나 네비가 안 보일 수 있음.

### 2.3 너비

| 항목 | 현재 | 문제 |
|------|------|------|
| 앱 컨테이너 | `w-[393px]` 고정 | 360px 등 더 좁은 기기에서 가로 스크롤/잘림 또는 레이아웃 깨짐. |

→ **원인**: 최소 너비 제한 없이 393px 고정이라, 작은 기기에서 전체가 안 들어갈 수 있음.

---

## 3. 개선 방향 (반응형 원칙)

1. **높이**: “한 화면 높이”를 **항상 실제 보이는 영역**으로 맞춘다.  
   → `100dvh` 우선, 미지원 시 `100vh` fallback.  
   → 컨테이너는 **고정 852px 제거**, `min-height: 100dvh` 등으로 “최소 한 화면”만 보장.
2. **레이아웃**: **본문 + 네비**를 하나의 **flex column**으로 묶고,  
   → **본문**: `flex: 1; min-height: 0; overflow: auto` (필요 시 스크롤).  
   → **네비**: `flex-shrink: 0` + `padding-bottom: env(safe-area-inset-bottom)`.  
   → 네비는 “항상 화면 하단에 고정”이 아니라 “레이아웃의 맨 아래 블록”으로 두어, 어떤 기기에서도 같은 컨테이너 안에 포함되게 한다.
3. **너비**: 393px는 “최대 너비”로 두고, 작은 기기는 **100%**로 채운다.  
   → `width: 100%; max-width: 393px`.
4. **safe area**: 노치/홈 인디케이터를 반영해 네비가 가려지지 않게 한다.  
   → `viewport-fit=cover` + 기존처럼 `env(safe-area-inset-bottom)` 유지·보강.

## 4. 단계별 작업 계획

### Phase 1: 뷰포트·메타·글로벌 스타일 (기반)

| 순서 | 작업 | 파일 | 내용 |
|------|------|------|------|
| 1.1 | 뷰포트 메타 보강 | `index.html` | `viewport-fit=cover` 추가. (선택) `user-scalable=no` 등은 필요 시만. |
| 1.2 | 루트 높이 통일 | `index.css` 또는 `App.tsx` 래퍼 | `html, body, #root`에 `height: 100%` 또는 `min-height: 100dvh` + `100vh` fallback. |
| 1.3 | 앱 컨테이너 높이/너비 | `App.tsx` | 내부 컨테이너: `h-[852px]` 제거 → `min-height: 100dvh`(또는 100vh), `height: 100%` 등으로 “한 화면 전체” 채우기. `w-[393px]` → `w-full max-w-[393px]`. |

**검증**: 다양한 기기에서 앱 영역이 화면 전체를 채우고, 세로 스크롤 없이 한 화면에 들어오는지 확인.

---

### Phase 2: 앱 레이아웃 구조 (본문 + 네비 항상 보이게)

| 순서 | 작업 | 파일 | 내용 |
|------|------|------|------|
| 2.1 | 앱 내부를 flex column으로 | `App.tsx` | 화면을 감싸는 영역을 `flex flex-col min-h-0 h-full` 구조로 변경. |
| 2.2 | 각 화면 공통 레이아웃 패턴 정의 | 새 레이아웃 컴포넌트 또는 각 Screen | 상단: StatusBar 등 고정 영역. 중간: `flex-1 min-h-0 overflow-auto`(본문). 하단: 네비 `flex-shrink: 0` + safe-area. |
| 2.3 | 네비게이션바 위치 방식 통일 | `NavigationBar.tsx` + 각 Screen | `position: fixed` 대신 **부모 flex 컨테이너의 맨 아래**로 두는 방식으로 변경. (부모가 `flex flex-col`, 네비는 마지막 자식.) |
| 2.4 | 본문 영역 하단 여백 | 각 Screen 또는 공통 레이아웃 | 본문 컨텐츠 하단에 네비 높이 + safe-area만큼 `padding-bottom` 또는 `margin-bottom` 부여해 스크롤 시 네비에 가리지 않게 함. |

**검증**: 짧은 기기·긴 기기 모두에서 하단 네비가 항상 보이고, 본문은 스크롤만 되고 잘리지 않는지 확인.

---

### Phase 3: 개별 화면 적용 (Record, Home, Report 등)

| 순서 | 작업 | 대상 | 내용 |
|------|------|------|------|
| 3.1 | RecordScreen | `RecordScreen.tsx` | 상단(StatusBar, TopBar) / 본문(스크롤 가능) / 네비 구조로 재구성. `absolute top-[96px]` 제거 후 flex 영역 안에 배치. |
| 3.2 | HomeScreen | `HomeScreen.tsx` | 동일하게 상단·본문·네비 구간 나누기. |
| 3.3 | ReportScreen, ReportListScreen, MyScreen 등 | 해당 tsx | 네비 있는 화면은 모두 같은 패턴 적용. |
| 3.4 | 네비 없는 화면 (Intro, Splash, Login 등) | 해당 tsx | 전체 높이만 채우도록 `min-height: 100dvh` 등 유지. |

**검증**: 탭 전환·스크롤 시 레이아웃 깨짐 없이 모든 내용이 보이는지 확인.

---

### Phase 4: safe area·특수 환경

| 순서 | 작업 | 파일 | 내용 |
|------|------|------|------|
| 4.1 | 상단 safe area | 공통 레이아웃 또는 StatusBar | `padding-top: env(safe-area-inset-top)` 적용해 노치 아래에 내용 시작. |
| 4.2 | 하단 safe area | `NavigationBar.tsx` | 이미 `padding-bottom: env(safe-area-inset-bottom)` 있음. 뷰포트가 `viewport-fit=cover`로 설정됐는지 확인. |
| 4.3 | iOS 주소창 | - | `100dvh` 사용으로 최소한의 개선. 필요 시 `svh`/`lvh`는 사용 목적에 따라 검토. |

**검증**: 노치 있는 iPhone, 홈 인디케이터 있는 Android에서 상·하단이 잘리지 않는지 확인.

---

### Phase 5: 정리·테스트

| 순서 | 작업 | 내용 |
|------|------|------|
| 5.1 | 사용하지 않는 스타일 제거 | 고정 `h-[852px]` 등 레거시 높이 클래스 정리. |
| 5.2 | 실제 기기 테스트 | iOS Safari, Android Chrome, 다양한 해상도(360x640, 393x852, 414x896 등)에서 확인. |
| 5.3 | 문서 반영 | 이 계획서를 “구현 완료 시” 결과로 업데이트하거나, 간단한 “반응형 가이드” 한 줄 요약 추가. |

---

## 5. 수정 시 주의사항

- **디자인 픽셀 기준(393x852)** 는 “기준 해상도”로만 두고, 실제 코드에서는 **비율·flex·%·dvh**로 바꾸는 것이 안전함.
- **고정 픽셀 높이**(예: 852px, 96px 상단 등)는 **최소 단위**만 유지하고, 전체 높이는 **dvh + flex**로 맞추는 것을 권장.
- 네비를 **fixed**로 두면 “뷰포트 하단”에 붙지만, 컨테이너가 `overflow: hidden`이면 기기마다 보이지 않을 수 있으므로, **flex로 항상 컨테이너 안에 두는 방식**이 더 예측 가능함.
